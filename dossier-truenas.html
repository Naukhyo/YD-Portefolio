TrueNAS Scale — Bras-d-Orion

Configuration d'un serveur NAS TrueNAS Scale, intégration au domaine Active Directory milkyway.local, et mise en place des partages SMB pour les utilisateurs et les GPO. Ce dossier couvre l'installation, la configuration réseau, la jonction AD et la gestion des partages.

Présentation du serveur

TrueNAS Scale est installé sur une machine physique dédiée dans la salle serveur — c'est pas une VM contrairement aux DCs. L'avantage c'est d'avoir du stockage physique direct, ce qui est plus adapté pour un NAS.

1. Configuration réseau et hostname

TrueNAS prend une IP en DHCP par défaut — on passe en IP fixe via Network → Interfaces et on configure le hostname dans Network → Global Configuration.

Paramètres Global Configuration

  • Hostname : Bras-d-Orion
  • Domain : milkyway.local
  • DNS Nameserver 1 : 192.168.10.3 (DC principal Milky-Way)
  • Default Gateway : 192.168.10.1
  • Inherit domain from DHCP : décoché
Point important sur le DNS : Il faut mettre uniquement le DC principal (192.168.10.3) en DNS. Si on met 8.8.8.8 en plus, TrueNAS essaie de résoudre milkyway.local via Google et la jonction AD échoue. Le DNS du domaine doit être le seul nameserver.

2. Jonction au domaine Active Directory

On joint TrueNAS au domaine milkyway.local via Credentials → Directory Services → Configure Active Directory. Ça permet de gérer les accès aux partages avec les groupes AD directement, sans recréer de comptes locaux.

Prérequis avant la jonction

  • L'IP fixe et le hostname sont bien configurés (voir section précédente)
  • Le DNS pointe uniquement sur 192.168.10.3 (le DC) — pas de 8.8.8.8 en plus
  • Le DC principal Milky-Way est accessible en ping depuis TrueNAS

3. Partages SMB

Les partages SMB permettent aux postes Windows d'accéder aux dossiers du NAS comme un lecteur réseau. Chaque partage est basé sur un dataset ZFS — ça permet de gérer les permissions séparément pour chacun. Étapes :

  1. 1Aller dans Storage → Create Pool (déjà fait) puis créer un dataset pour chaque partage dans Storage → Datasets
  2. 2Configurer les ACL du dataset pour donner les bons accès aux groupes AD (GRP-Informatique, etc.)
  3. 3Créer le partage SMB dans Shares → Windows (SMB) en pointant sur le dataset
  4. 4Démarrer le service SMB si ce n'est pas déjà fait (état "Running" requis)

Gestion des ACL

TrueNAS utilise des ACL POSIX sur les datasets ZFS pour gérer les permissions. On les configure via Storage → Datasets → Edit Permissions. Puisque TrueNAS est joint au domaine, on peut directement ajouter les groupes AD dans les ACL.

GRP-Utilisateurs n'a accès ni au partage Informatique ni au partage Direction. GRP-Direction n'a accès qu'à son propre partage. Seul GRP-Informatique a accès au partage Commun et au partage Informatique.

Lien avec la GPO-FondEcran

Le partage GPO est lié à la GPO-FondEcran configurée sur Milky-Way. Le chemin renseigné dans la GPO est \\Bras-d-Orion\GPO\fond-ecran.jpg — à chaque ouverture de session, Windows récupère l'image depuis le NAS automatiquement. Pour que ça marche il faut que TrueNAS soit joint au domaine (statut HEALTHY), que le service SMB soit actif, et que le fichier image soit déposé dans le dataset GPO.

SMB v1 désactivé

SMB v1 est un ancien protocole de partage de fichiers qui date des années 90. Il est désactivé sur TrueNAS par défaut, et c'est une bonne chose — SMB v1 a été exploité par le ransomware WannaCry en 2017 pour se propager sur les réseaux d'entreprise. Le protocole ne chiffre pas les données et n'a pas de mécanisme d'authentification moderne.

  • SMB v1 : désactivé — protocole vulnérable, à ne jamais réactiver
  • SMB v2 et v3 : activés — chiffrement des données en transit, authentification Kerberos
  • TrueNAS Scale désactive SMB v1 par défaut dans les paramètres du service SMB
Tous les postes Windows 10/11 supportent SMB v2 et v3 nativement. La désactivation de SMB v1 n'a aucun impact sur les utilisateurs — ils accèdent aux partages exactement pareil.
