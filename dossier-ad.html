Active Directory — milkyway.local

Mise en place d'une infrastructure réseau complète avec Active Directory sous Windows Server 2022. Ce projet couvre l'installation et la configuration d'un DC principal, d'un DC secondaire pour la redondance, la gestion des utilisateurs et groupes, la configuration DNS/DHCP, et la mise en place de GPO de sécurité.

Infrastructure globale

Voici un récapitulatif de toutes les machines impliquées dans ce projet. Tout tourne sur Proxmox — les deux serveurs Windows sont des VMs. Le domaine s'appelle milkyway.local. J'ai utilisé un thème astronomique pour nommer les machines, c'est plus pratique pour les distinguer.

1. Installation et configuration de base

Avant de faire quoi que ce soit avec l'AD, il faut d'abord faire les mises à jour Windows et les drivers. Sur une VM Proxmox fraîchement installée, il y a souvent des pilotes manquants ou des patchs de sécurité en attente — mieux vaut régler ça avant de commencer à configurer quoi que ce soit. Ça évite les redémarrages intempestifs en pleine config et les comportements bizarres liés à un OS pas à jour.

Une fois les mises à jour faites et le serveur redémarré, on peut configurer le réseau. Sur une VM Proxmox, Windows Server 2022 démarre avec une adresse IP en DHCP. Il faut passer en IP fixe avant d'installer les rôles.

La configuration réseau se fait dans Panneau de configuration → Centre Réseau et partage → Modifier les paramètres de la carte → Propriétés de l'adaptateur → IPv4. On renseigne les valeurs suivantes :

Adresse IP    : 192.168.10.3

Masque       : 255.255.255.0 (/24)

Passerelle   : 192.168.10.1 (pfSense)

DNS préféré  : 127.0.0.1 (lui-même — obligatoire pour un DC)

Propriétés TCP/IPv4 — IP fixe 192.168.10.3, DNS préféré 127.0.0.1 (lui-même), auxiliaire 8.8.8.8

Important — DNS sur 127.0.0.1 : Le DNS doit pointer sur lui-même (127.0.0.1) sur le DC principal. Si on met une autre IP, la promotion en contrôleur de domaine peut échouer.

Une fois l'IP fixe configurée, il faut renommer l'ordinateur en Milky-Way via les Propriétés système (clic droit sur "Ce PC" → Propriétés → Modifier les paramètres → Modifier). Le nom doit correspondre à ce qu'on veut voir dans l'AD. Après le renommage, Windows demande un redémarrage — il faut le faire avant de continuer.

Après le redémarrage, on peut vérifier la configuration réseau en ouvrant une invite de commandes et en tapant ipconfig /all. On doit voir l'IP 192.168.10.3 avec le DNS à 127.0.0.1. Si c'est bon, on peut passer à l'installation des rôles.

2. Installation du rôle AD DS

L'installation du rôle AD DS se fait via le Gestionnaire de serveur → Gérer → Ajouter des rôles et fonctionnalités. C'est un assistant en plusieurs étapes, assez guidé. On choisit d'installer plusieurs rôles en même temps pour gagner du temps.

  1. Type d'installation : choisir "Installation basée sur un rôle ou une fonctionnalité"
  2. Sélection du serveur : SRV-AD (Milky-Way) — le serveur local apparaît dans la liste
  3. Rôles sélectionnés : Services AD DS + Serveur DHCP + Serveur DNS — les trois cochés en même temps
  4. Fonctionnalités : laisser les fonctionnalités par défaut proposées par l'assistant
  5. Confirmation et installation : cliquer sur Installer et attendre la fin (ça prend quelques minutes)

Sélection des rôles — AD DS, DHCP, DNS cochés dans l'assistant

Détail du rôle AD DS — Services de domaine Active Directory

Une fois l'installation terminée, une notification apparaît dans le Gestionnaire de serveur (le drapeau en haut à droite avec un triangle d'alerte) pour promouvoir le serveur en contrôleur de domaine. C'est l'étape suivante.

Note : Si la notification n'apparaît pas tout de suite, rafraîchir le Gestionnaire de serveur avec F5. Parfois il y a un petit délai après l'installation des rôles.

3. Promotion en contrôleur de domaine

La promotion se lance via la notification "Promouvoir ce serveur en contrôleur de domaine" dans le Gestionnaire de serveur. L'assistant demande plusieurs informations importantes — c'est là qu'on définit le nom du domaine et les paramètres fondamentaux de la forêt AD.

Paramètres de promotion saisis

  • Opération de déploiement : Ajouter une nouvelle forêt (c'est la première fois qu'on crée ce domaine)
  • Nom du domaine racine : milkyway.local
  • Niveau fonctionnel de la forêt : Windows Server 2016
  • Niveau fonctionnel du domaine : Windows Server 2016
  • Options du DC : Serveur DNS activé, Catalogue global activé
  • Mot de passe DSRM : défini (nécessaire pour restaurer l'AD en mode de restauration des services d'annuaire)
  • Chemin de la base NTDS, SYSVOL, journaux : laissés par défaut (C:\Windows\NTDS)
Pourquoi Windows Server 2016 comme niveau fonctionnel ? On pourrait choisir Windows Server 2022 mais rester sur 2016 permet d'ajouter des DCs avec des versions antérieures si besoin. Dans ce projet c'est pas critique, mais c'est une bonne pratique.

Avant de lancer la promotion, l'assistant fait une vérification de prérequis. Si tout est vert (sauf éventuellement un avertissement non bloquant sur le mot de passe admin), on peut cliquer sur Installer. Le serveur redémarre automatiquement après la promotion.

Au redémarrage, le domaine milkyway.local est créé et Milky-Way est le DC principal. On se connecte maintenant avec le compte MILKYWAY\Administrateur (et non plus juste Administrateur).

4. Configuration DNS

Le DNS est essentiel pour l'AD — sans DNS qui fonctionne, rien ne marche. Quand on dit "rien", c'est vraiment rien : les postes ne peuvent pas trouver le DC, les utilisateurs ne peuvent pas se connecter au domaine, les GPO ne s'appliquent pas. C'est le service le plus critique de toute l'infrastructure AD.

Le rôle DNS est installé en même temps qu'AD DS et une zone DNS principale est automatiquement créée pour milkyway.local. La console DNS est accessible via Outils → DNS dans le Gestionnaire de serveur.

Comment le DNS est structuré dans l'AD

Quand on promeut Milky-Way en DC, Windows crée automatiquement une zone DNS principale pour milkyway.local. Cette zone contient tous les enregistrements dont l'AD a besoin pour fonctionner. Il y a deux types d'enregistrements importants :

  • Enregistrements A : associent un nom à une IP. Le DC s'enregistre automatiquement : milky-way.milkyway.local → 192.168.10.3. Idem pour Andromède une fois prêt.
  • Enregistrements SRV : enregistrements de services, visibles dans le sous-domaine _msdcs.milkyway.local. Ils permettent aux clients de localiser les DCs, les serveurs Kerberos, etc. Sans eux, les postes ne peuvent pas rejoindre le domaine.
  • Zone de recherche inversée : créée pour le réseau 192.168.10.x — permet de résoudre une IP en nom. Utile pour les logs et certains outils de diagnostic.
  • Redirecteurs : configurés vers 8.8.8.8 (Google DNS) pour la résolution des noms externes (internet). Les requêtes milkyway.local restent traitées localement, les autres sont transmises à Google.

Points vérifiés après installation

  • Zone de recherche directe milkyway.local présente dans la console DNS
  • Enregistrements SRV créés automatiquement par AD (visibles dans _msdcs.milkyway.local)
  • Le DC s'est enregistré automatiquement : milky-way.milkyway.local → 192.168.10.3
  • Zone de recherche inversée créée pour 192.168.10.x
  • Redirecteurs configurés vers 8.8.8.8 pour la résolution externe
Vérification DNS : Pour tester que le DNS fonctionne, ouvrir un cmd et taper nslookup milkyway.local. On doit obtenir l'IP 192.168.10.3 en réponse. Si le résultat dit "serveur inconnu" ou "impossible de trouver", le DNS ne fonctionne pas et l'AD ne marchera pas correctement.

Pourquoi le DNS est critique pour l'AD

L'Active Directory dépend totalement du DNS. Si le DNS plante ou est mal configuré, rien ne marche : les postes ne trouvent plus le DC, les utilisateurs ne peuvent plus s'authentifier, les GPO ne s'appliquent plus. C'est le service le plus critique de toute l'infrastructure. C'est pour ça que le DC doit être son propre serveur DNS et que les clients doivent avoir l'IP du DC en DNS primaire.

Les clients du domaine reçoivent automatiquement 192.168.10.3 comme serveur DNS via le DHCP (voir section suivante). C'est l'option 006 de l'étendue DHCP.

5. Configuration du DHCP

Le rôle DHCP est installé sur Milky-Way en même temps qu'AD DS et DNS. Son rôle c'est de distribuer automatiquement les adresses IP aux postes clients qui rejoignent le réseau, ce qui évite de devoir tout configurer manuellement. La console DHCP est accessible via Outils → DHCP dans le Gestionnaire de serveur.

Étape 1 — Autoriser le serveur DHCP dans l'AD

C'est une étape qu'on oublie souvent : avant de créer une étendue, il faut autoriser le serveur DHCP dans l'AD. Dans un environnement Active Directory, un serveur DHCP non autorisé ne distribue aucune adresse IP, même s'il est correctement configuré. Ça se fait via un clic droit sur le serveur dans la console DHCP → Autoriser. Le serveur passe alors de l'état rouge (non autorisé) à l'état vert.

Pourquoi cette autorisation ? C'est une mesure de sécurité Microsoft : si quelqu'un branche un serveur DHCP non autorisé sur le réseau, il ne pourra pas distribuer des adresses qui entreraient en conflit avec le DHCP officiel du domaine. Ça protège contre les "rogue DHCP".

Étape 2 — Créer l'étendue DHCP

Une fois le serveur autorisé, on crée l'étendue via clic droit sur "IPv4" dans la console → Nouvelle étendue. L'assistant demande le nom de l'étendue, la plage d'adresses, les exclusions et les options (passerelle, DNS).

Paramètres de l'étendue DHCP — LAN-10

Paramètre Valeur
Nom de l'étendue LAN-10
Plage d'adresses 192.168.10.100 → 192.168.10.200
Masque de sous-réseau 255.255.255.0
Passerelle (option 003) 192.168.10.1 — pfSense
Serveur DNS (option 006) 192.168.10.3 — Milky-Way
Suffixe DNS (option 015) milkyway.local
Durée du bail 8 jours

Les exclusions sont configurées pour les IPs fixes : 192.168.10.1 à 192.168.10.99 sont exclus de la plage DHCP pour éviter les conflits avec les serveurs et équipements réseau. Les serveurs, le pare-feu et le NAS ont tous des IPs fixes dans cette plage.

Pourquoi 8 jours de bail ? Dans un réseau d'entreprise, 8 jours c'est un bon compromis. Un poste qui reste allumé et connecté garde son IP. Un poste qui ne répond plus depuis 8 jours voit son IP libérée pour être redistribuée. Trop court (1 jour) → les IPs changent tout le temps et ça peut casser des configs. Trop long (30 jours) → les IPs ne se libèrent jamais et on finit à court.

Ce que les postes reçoivent automatiquement

Quand un poste Windows démarre et obtient une adresse DHCP, il reçoit en même temps plusieurs informations via les options :

  • Une IP dans la plage 192.168.10.100–200 — assignée dynamiquement par le DHCP
  • La passerelle par défaut (option 003) : 192.168.10.1 — pfSense, pour sortir sur internet
  • Le serveur DNS (option 006) : 192.168.10.3 — Milky-Way, pour résoudre les noms du domaine
  • Le suffixe DNS (option 015) : milkyway.local — le poste peut alors résoudre directement milky-way au lieu de milky-way.milkyway.local

C'est grâce à ces options que les postes peuvent rejoindre le domaine et s'authentifier automatiquement dès qu'ils démarrent sur le réseau.

6. Organisation de l'annuaire — OUs, groupes et comptes

Une fois le domaine créé, on organise l'annuaire Active Directory avec des Unités d'Organisation (OUs). C'est ce qui permet de structurer les comptes et d'y appliquer des GPO de façon ciblée. Sans OUs, tous les comptes sont dans le même conteneur par défaut et on ne peut pas cibler les GPO précisément.

La console utilisée pour toute cette partie c'est ADUC (Active Directory Users and Computers), accessible via Outils dans le Gestionnaire de serveur.

OUs créées

Les OUs se créent par un clic droit sur le domaine (milkyway.local) → Nouveau → Unité d'organisation. J'ai créé l'OU avec l'option "Protéger le conteneur de la suppression accidentelle" cochée — bonne pratique pour éviter de supprimer une OU par erreur.

  • OU-Direction : contient les comptes de la direction
  • OU-Informatique : contient les comptes du service informatique
  • OU-Ordinateurs : pour les postes clients (quand ils joignent le domaine)
  • OU-Utilisateurs : conteneur général pour les autres utilisateurs

Création d'une OU via clic droit sur milkyway.local → Nouveau → Unité d'organisation

Groupes créés

Les groupes permettent de gérer les droits d'accès aux ressources de façon centralisée. On ajoute les utilisateurs dans les groupes, et on donne les droits aux groupes — pas directement aux utilisateurs. C'est plus facile à maintenir.

Groupe Type Périmètre Usage
GRP-Direction Sécurité Global Accès aux ressources de la Direction
GRP-Informatique Sécurité Global Accès aux ressources du service Informatique

7. Mise en place du DC secondaire (Andromède)

Pour avoir de la redondance, j'ai ajouté une deuxième VM Windows Server 2022 appelée Andromède (IP 192.168.10.4). L'idée c'est que si Milky-Way tombe, les utilisateurs peuvent toujours s'authentifier via Andromède. Sans DC secondaire, si le DC principal plante, personne ne peut se connecter au domaine.

Étapes de mise en place

  1. Installer Windows Server 2022 sur la VM Andromède dans Proxmox (même procédure que pour Milky-Way)
  2. Configurer l'IP fixe : 192.168.10.4, masque 255.255.255.0, passerelle 192.168.10.1
  3. DNS : pointer sur 192.168.10.3 (Milky-Way) — sur le DC secondaire, le DNS pointe sur le DC principal, pas sur lui-même
  4. Renommer la machine en Andromède via les Propriétés système, puis redémarrer
  5. Joindre la machine au domaine milkyway.local (Propriétés système → Modifier → Domaine → entrer milkyway.local + credentials admin)
  6. Installer le rôle AD DS via le Gestionnaire de serveur
  7. Promouvoir en DC secondaire via la notification dans le Gestionnaire de serveur

Gestionnaire de serveur sur Andromède — la notification de promotion en DC est disponible après l'installation d'AD DS

Promotion en DC secondaire

La promotion d'Andromède en DC secondaire se fait en choisissant "Ajouter un contrôleur de domaine à un domaine existant". Il faut fournir les credentials d'un admin du domaine (ANDROMEDE\Administrateur dans ce cas). L'assistant vérifie que le domaine milkyway.local est accessible depuis Andromède avant de continuer.

Assistant de promotion — option "Ajouter un contrôleur de domaine à un domaine existant" sélectionnée pour milkyway.local

Options du DC secondaire — DNS et Catalogue global activés, mot de passe DSRM défini

Après le redémarrage d'Andromède, les deux DCs sont visibles dans la console ADUC sous Domain Controllers : MILKY-WAY et ANDROMEDE. La réplication AD se fait automatiquement entre les deux — les modifications faites sur Milky-Way (nouveaux utilisateurs, nouvelles GPO...) sont répliquées sur Andromède en quelques secondes.

Vérification de la réplication : Pour tester que la réplication fonctionne, on peut ouvrir un cmd sur Andromède et taper repadmin /showrepl. Cette commande affiche l'état de la réplication entre les DCs. Si tout est OK, on doit voir "Aucune erreur" (ou 0 échec de réplication).

8. Configuration des GPO

Les GPO (Group Policy Objects) permettent d'appliquer des paramètres de configuration à tous les utilisateurs et ordinateurs du domaine depuis un point central. On les gère depuis la console Gestion des stratégies de groupe (GPMC), accessible via Outils dans le Gestionnaire de serveur. J'en ai créé 4 pour sécuriser l'environnement.

Comment créer une GPO

Dans la GPMC, on fait un clic droit sur le domaine ou l'OU où on veut la GPO → Créer un objet GPO dans ce domaine, et le lier ici.... Une fenêtre s'ouvre pour donner un nom à la GPO. Une fois créée, on double-clique dessus pour ouvrir l'éditeur et configurer les paramètres.

Clic droit sur milkyway.local pour créer une nouvelle GPO

Saisie du nom — GPO-PasswordPolicy

Exemple de création d'une GPO — ici GPO-Securite, même procédure pour toutes les GPO du domaine

Une fois la GPO créée, elle apparaît dans la liste des GPO liées à l'OU ou au domaine. Elle peut être modifiée à tout moment en double-cliquant dessus pour ouvrir l'éditeur de stratégie de groupe.

Console GPMC — GPO-PasswordPolicy visible dans les GPO liées à milkyway.local

Vue générale de la console GPMC avec les GPO du domaine

GPO-PasswordPolicy

C'est la GPO la plus importante — elle définit les règles de mot de passe pour tous les utilisateurs du domaine. Elle est liée directement à la racine du domaine (milkyway.local) pour s'appliquer à tout le monde. La politique de mot de passe se configure dans Configuration ordinateur → Paramètres Windows → Paramètres de sécurité → Stratégies de compte → Stratégie de mot de passe.

Les 3 autres GPO

En plus de la politique de mot de passe, j'ai créé 3 GPO pour restreindre les actions des utilisateurs standards et standardiser l'environnement de travail.

GPO-Restriction

Liée à OU-Utilisateurs. Bloque l'accès au Panneau de configuration pour les utilisateurs standards. Configurée dans Configuration utilisateur → Modèles d'administration → Panneau de configuration → Interdire l'accès au Panneau de configuration et à PC Settings.

GPO-Application

Liée à OU-Utilisateurs. Bloque l'accès à l'invite de commandes (cmd.exe) pour les utilisateurs. Configurée dans Configuration utilisateur → Modèles d'administration → Système → Désactiver l'accès à l'invite de commandes.

GPO-FondEcran

Impose un fond d'écran commun à tous les utilisateurs du domaine. Le fichier image est stocké sur le partage \\Bras-d-Orion\GPO\ (TrueNAS). GPO configurée, en attente des partages SMB.

GPO-PasswordPolicy

Liée à milkyway.local (domaine entier). Politique de mot de passe : 8 caractères min, complexité activée, durée max 90 jours, verrouillage après 5 tentatives / 10 minutes.

GPO-Application — Blocage de cmd.exe

Pour la GPO qui bloque l'invite de commandes, on va dans l'éditeur → Configuration utilisateur → Modèles d'administration → Système → et on double-clique sur "Désactiver l'accès à l'invite de commandes". On passe le paramètre sur "Activé".

GPO-Application — paramètre "Désactiver l'accès à l'invite de commandes" dans l'éditeur de stratégie

Arborescence de l'éditeur GPO — on voit les deux sections Configuration ordinateur et Configuration utilisateur

Comment appliquer une GPO à une OU spécifique ? Dans la GPMC, on fait un clic droit sur l'OU concernée → Lier un objet de stratégie de groupe existant, puis on choisit la GPO. GPO-Restriction et GPO-Application sont liées à OU-Utilisateurs pour ne pas bloquer les administrateurs. GPO-PasswordPolicy est liée au domaine entier car la politique de mot de passe doit s'appliquer à tous.
